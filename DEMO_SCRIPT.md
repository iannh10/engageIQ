## Q&A — prepped answers

### Technical questions

- **"Walk me through how the ranking actually works."**
  "The profile and each opportunity are both embedded using Sentence-BERT
  (`all-MiniLM-L6-v2`, 384 dimensions). Cosine similarity between them becomes the
  relevance score — 45% of the final score. That gets blended with community health
  (15%), visibility (15%), freshness (10%), effort-fit (10%), and feedback affinity (20%).
  Then an avoid filter strips excluded topics, and a diversification pass spreads the top
  10 across domains."

- **"Why is your Precision@10 so high — is this overfitting?"**
  "That's a fair challenge. The simulation defines relevance using the same signals the
  ranker already uses — persona interest tokens, tags, and text — so the evaluation is
  somewhat circular. David's 0.9→1.0 shows genuine improvement, but I'd describe it as a
  *consistency check* that the pipeline responds correctly to persona signals, not as proof
  of real-world generalization accuracy. A stricter test would use manually labeled holdout
  data or hard-negative examples."

- **"How does SBERT embed a user profile? It's not a sentence."**
  "The profile's interest string — e.g., 'machine learning, Python, beginner portfolio
  projects' — is passed directly to the MiniLM encoder as text. The encoder was trained on
  diverse sentence pairs, so it produces a meaningful semantic vector even for comma-
  separated keyword lists."

- **"What does the diversification step do?"**
  "After scoring, results from the same domain are slightly penalized to avoid the top 10
  being all Machine Learning or all GitHub. It spreads coverage across domains so the
  user gets a broader weekly action list."

- **"What's the difference between Engage and Bookmark?"**
  "Both are positive signals, but Engage (+0.30) is stronger than Bookmark (+0.20) — it
  signals you *acted*, not just saved. Skip is −0.30. These deltas accumulate on topic
  tags so the system shifts its weights toward what the user consistently acts on."

- **"SBERT vs TF-IDF — which is running on the live site?"**
  "The public deploy uses TF-IDF cosine retrieval — the 113 MB SBERT cache isn't shipped
  to GitHub since it's above the file size limit. SBERT runs locally where the cache
  exists. Same ranking pipeline either way; SBERT just produces richer semantic vectors."

- **"How does the avoid filter work technically?"**
  "The avoid string is split on commas and semicolons. Any opportunity whose title or tags
  contain one of those terms (case-insensitive, tokens over 3 characters) is filtered out
  before scoring. It runs on every rerank, so edits to the box take effect immediately."

### Conceptual / grading questions

- **"Why DEV.to instead of Reddit?"**
  "Reddit's Data API approval is a multi-week process that couldn't be completed in the
  project timeline. Forem/DEV.to is an equivalent public developer-community platform
  covering all 15 domains and satisfies the same rubric intent. It's documented in the
  brief's limitations section."

- **"Is the data real or synthetic?"**
  "It's a schema-compatible hybrid: a deterministic synthetic baseline of 700 records per
  domain, plus live-ingested records from GitHub, GH Archive, Hacker News, and Forem.
  Live ingestion was tested end-to-end; Forem runs unauthenticated."

- **"Does the custom profile actually work differently from the preset personas?"**
  "Yes — the same embedding and scoring pipeline runs against whatever text you enter.
  Different interests produce different cosine similarities across the 76k opportunities,
  so the ranked output is genuinely different, not just relabeled."

- **"What would make this production-ready?"**
  "Three things: a real holdout evaluation set with manually labeled relevance (to replace
  the circular simulation), per-user persistent storage (right now feedback_events.json is
  shared across users), and live daily ingestion with incremental embedding updates instead
  of a static snapshot."

- **"Why Streamlit?"**
  "Speed of iteration and zero frontend code — the whole dashboard is Python. For a
  research prototype that needs to be live within a project timeline, it's the right
  trade-off. The rubric also allows it explicitly."

## Timing & contingency

- **Practice the 60-round sim once** — it's your highest-value moment. Know how long it
  takes so you don't talk into dead air.
- If the **LLM suggestion is slow/blank**, say "this calls Claude live, so latency depends
  on the API" and move on — the fallback action still shows.
- 1-minute buffer is built in. If you run long, **cut the custom-profile demo (4:30–5:45)
  first** — it's the most expendable segment.
- If asked about precision being too high, **don't be defensive** — Codex flagged this
  same concern during testing. Own the limitation: "it's a consistency check, not a
  generalization proof." That reads as intellectually honest, not weak.
