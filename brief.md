## 1. System Overview

EngageIQ is a smart engagement opportunity scorer that ingests public records from GitHub, GH Archive, Forem (DEV.to) articles, and Hacker News, then ranks them for a professional deciding where to invest time online. The deployed Streamlit dashboard lets a user switch between the four test personas or build a custom profile, then surfaces ranked opportunities with "Why this?" explanations, LLM-generated engagement suggestions (Claude Haiku), inline engage / bookmark / skip feedback, and CSV/PDF brief downloads.

The current offline grading dataset contains **76,972 unique records** across **all 15 required technical domains** and **4 sources**, materially exceeding the 10,000-record requirement. The dataset is generated by `code/engageiq/data_generator.py` as a deterministic 10,500-record baseline (700 records per domain), then expanded by live ingestion via `code/ingest_live.py` against the GitHub REST API, GH Archive, the Hacker News API, and the Forem (DEV.to) public articles API. The Forem collector requires no credentials, so the snapshot ships with Forem community records.

**Deduplication.** Every record carries a `stable_id` derived from `sha1(source | title | url)`. Offline generation collects records into a dictionary keyed by id; live append goes through `pandas.drop_duplicates(subset=["id"])`. Both paths converge on identity-based dedup so duplicates collapse before scoring.

**Streaming caveat.** Live ingestion is currently invoked as a batch script rather than a continuously running stream. The same `normalize_record` schema and dedup mechanism would extend cleanly to a polling or webhook driver — flagged as a limitation in §6.

## 3. BAX-423 Techniques

EngageIQ integrates three techniques from the course and benchmarks the matching/ranking layer.

**Technique 1 — Semantic embedding retrieval (Sentence-BERT).**
Each opportunity is represented by the concatenation of `title + body + domain + tags`. Each user profile is represented by `name + interests + goal + skillsets + avoid`. Both are encoded with `sentence-transformers/all-MiniLM-L6-v2` and normalized to unit length. The opportunity matrix is cached in `data/engageiq_sbert_embeddings.npy` (≈118 MB) with metadata in the sibling `.json`, so the dashboard reuses the embeddings after the first build instead of recomputing 77k vectors per startup. Cosine similarity (a single matrix-vector multiply on normalized vectors) returns the top-K candidate set before the scoring layer runs. If `sentence-transformers` is unavailable in a grading environment, the ranker transparently falls back to TF-IDF cosine retrieval so the application remains runnable.

**Technique 2 — Multi-stage recommendation and ranking.**
Candidate generation retrieves the top-500 semantically related records. A scoring layer then computes a composite engagement score:

```
final_score = 45·relevance + 15·health + 15·visibility
            + 10·freshness + 10·effort_fit + 5·feedback_affinity
```

Health combines activity, comments, and contributors minus toxicity. Visibility combines platform score, stars, and growth rate. Effort fit penalizes opportunities far from the persona's weekly time budget. Persona-specific boosts add trend-velocity weight for the journalist profile, good-first-issue weight for beginner-oriented profiles, infra-domain weight for DevOps, and developer-tools weight for the startup founder. A final diversification pass penalizes repeated domains and over-represented sources so the top-10 does not collapse into a narrow slice of the dataset.

**Technique 3 — Adaptive learning from feedback.**
Engage / bookmark / skip events translate into tag-level weight deltas (+0.08, +0.05, −0.07) that feed back into `feedback_affinity` on the next render. Real dashboard clicks are persisted to `data/feedback_events.json` and replayed into the ranker at startup via `FeedbackStore.apply_to_ranker`, so learned preferences survive restarts. The simulation in `code/engageiq/learning.py` runs 60 rounds of engage/skip choices per persona and reports Precision@10 before and after adaptation.

## 4. Benchmarks

**Retrieval backend benchmark (`code/evaluate_personas.py`).** The same scoring pipeline is run with the SBERT dense backend and with the TF-IDF fallback. Precision@10 is computed from persona-interest token overlap on the top-10 titles, bodies, and tags.

| Persona                | TF-IDF P@10 | SBERT P@10 (before) | SBERT P@10 (after 60 rounds) | Δ |
|------------------------|------------:|--------------------:|------------------------------:|-----:|
| Sofia ML Student       | 1.00 | 1.00 | 1.00 | +0.00 |
| David DevOps Engineer  | 0.90 | 0.90 | 1.00 | +0.10 |
| Lina Data Journalist   | 1.00 | 1.00 | 1.00 | +0.00 |
| Raj Startup Founder    | 1.00 | 1.00 | 1.00 | +0.00 |

SBERT and TF-IDF achieve similar Precision@10 on this offline metric because three of four personas already saturate the metric. The SBERT embedding nevertheless produces materially better candidate ordering for off-vocabulary queries observed in the dashboard (e.g., a custom profile mentioning "observability tooling" retrieves Kubernetes/SRE records even when the literal token is absent). Adaptive learning lifts David from 0.90 → 1.00, the only persona below ceiling at start. The ceiling effect is a known weakness of the simulated relevance signal and is addressed in §6.

## 6. Limitations and Future Work

**Limitations.**
- The offline snapshot is schema-compatible with live API records but is partly synthetic. Live ingestion was tested end-to-end against GitHub, GH Archive, Hacker News, and the Forem (DEV.to) public articles API; the Forem path needs no credentials, so it runs unauthenticated against `https://dev.to/api/articles`.
- The project specification lists Reddit/PRAW as one of the four data sources. Reddit was replaced with Forem (DEV.to) because Reddit Data API approval is a multi-week process that could not be completed within the project timeline. Forem provides equivalent public developer-community discussions across all 15 domains and satisfies the same rubric intent: a community discussion platform ingested alongside GitHub and Hacker News.
- The pipeline is batch-driven by `ingest_live.py` rather than a continuously running stream.
- The offline Precision@10 metric ceilings at 1.00 for three of four personas because simulated relevance is derived from persona-interest token overlap on the same text used for retrieval. A held-out hand-labeled relevance set or NDCG@10 would be a stronger metric and is the highest-value next step.
- Suggested actions are template-generated from source + domain rather than LLM-generated.
- Streamlit Cloud's filesystem is ephemeral; `feedback_events.json` writes survive within a session but reset on container restart. The persistence path is correct for local runs and migrates cleanly to a managed database.
- Live ingestion has a domain-inference bias: the `infer_domain` helper in `collectors.py` over-attributes GH Archive events to Embedded Systems and Developer Tools because of a permissive keyword list. The synthetic baseline (10,500 records, 700 per domain) is unaffected.

**Future improvements.**
- Replace the batch ingestion script with scheduled GH Archive polling and webhook listeners.
- Add a held-out relevance set and replace Precision@10 with NDCG@10 to remove the ceiling effect.
- ~~Swap template suggested actions for an LLM call~~ **Implemented:** per-card "Suggested Action" is now generated by Claude Haiku (`claude-haiku-4-5`) using the opportunity title, platform, domain, and persona profile. Results are cached per (opportunity, persona) pair; falls back to a rule-based template when `ANTHROPIC_API_KEY` is not set.
- Migrate adaptive learning from tag-weight deltas to a contextual bandit so exploration vs exploitation becomes explicit.
- Add authenticated user accounts so multi-user feedback can be separated and survive deploys.
