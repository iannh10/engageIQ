# EngageIQ Methodology Documentation

## Project Goal

EngageIQ is a smart engagement opportunity scorer for the BAX-423 final project. It helps a professional decide where to invest time online by ranking public opportunities from GitHub, GH Archive, Forem (DEV.to) articles, and Hacker News-style posts.

The app is designed around the required workflow:

```text
multi-source ingestion -> deduplication -> content representation -> similarity retrieval
-> engagement scoring -> adaptive ranking -> trend analytics -> downloadable brief
```

## Data Pipeline

The offline grading dataset is generated by `code/engageiq/data_generator.py` and stored at:

```text
data/engageiq_opportunities.csv
```

The dataset contains 10,500 records, with 700 records in each of the 15 required technical domains. It includes four source labels:

- `github`
- `gh_archive`
- `forem`
- `hacker_news`

Each record has a stable `id`, source, domain, community, title, body, URL, author, timestamp, activity signals, growth signals, estimated effort, language, tags, and good-first-issue flag.

Deduplication is implemented by stable opportunity id. During snapshot generation, records are collected into a dictionary keyed by `id`, so duplicates collapse before writing the CSV. Optional live collectors in `code/engageiq/collectors.py` normalize public API records into the same schema and append with duplicate removal.

Live API ingestion can be run with:

```bash
python3 code/ingest_live.py --sources github gh_archive hacker_news
```

Forem (DEV.to) ingestion requires no credentials:

```bash
python3 code/ingest_live.py --sources forem --forem-tags machinelearning devops python
```

The live ingestion script collects public data only. It does not inspect private account history, saved posts, votes, messages, or private repositories.

## Submission Data Requirement

The EngageIQ instructions require an offline dataset large enough for grading and hidden-persona evaluation. The safest full-credit target is:

- at least 10,000 structured records
- coverage across all 15 technical domains
- an offline snapshot in the `data/` folder so graders can run without API access
- multiple public source categories such as GitHub, GH Archive, Forem (DEV.to), and Hacker News

The current local dataset exceeds this threshold. After live ingestion tests, `data/engageiq_opportunities.csv` contains 76,972 unique records across all 15 domains. Source counts in the current snapshot are:

- `gh_archive`: 69,092
- `github`: 2,630
- `forem`: 2,625
- `hacker_news`: 2,625

The original deterministic offline baseline was 10,500 records, with 700 records in each domain. Live ingestion appends additional public records and then applies exact `id` deduplication.

## Content Representation and Retrieval

The ranking layer lives in:

```text
code/engageiq/ranking.py
```

Each opportunity is represented as searchable text by combining:

- title
- body
- domain
- tags

The user profile is represented as text from:

- persona name
- interests
- goal
- skillsets
- avoid terms

EngageIQ uses a two-level retrieval design. The primary implementation uses Sentence-BERT dense embeddings with `sentence-transformers/all-MiniLM-L6-v2`. Each opportunity is encoded from its title, body, domain, and tags. Each user profile is encoded from the name, interests, goals, skillsets, and avoid terms. The profile embedding retrieves semantically related opportunities through cosine similarity before the multi-stage scoring pipeline ranks them by relevance, health, visibility, freshness, effort fit, and feedback affinity.

The dense opportunity matrix is cached in `data/engageiq_sbert_embeddings.npy`, with metadata in `data/engageiq_sbert_embeddings.json`. This avoids recomputing 10,000+ embeddings on every dashboard start. If SBERT or the model cache is unavailable in a grader environment, the app falls back to TF-IDF cosine retrieval so the application remains runnable.

## Engagement Scoring

After retrieving candidate opportunities, EngageIQ computes a composite score using:

- semantic relevance
- community health
- visibility potential
- freshness
- effort fit
- feedback affinity

Community health combines activity, comments, contributors, and toxicity. Visibility combines platform score, stars, and growth rate. Effort fit compares the opportunity's estimated minutes against the persona's weekly time budget.

The final scoring formula is:

```text
45 * relevance
+ 15 * health
+ 15 * visibility
+ 10 * freshness
+ 10 * effort_fit
+  5 * feedback_affinity
```

Persona-specific boosts are applied for trend-seeking, beginner-friendly contribution, DevOps/Kubernetes profiles, and startup-founder/developer-tool profiles.

## Re-Ranking and Diversity

After scoring, the app applies a diversity-aware re-ranking pass. It penalizes repeated domains and over-represented sources so the top results do not collapse into a narrow set of similar items.

The dashboard displays the final diversified score alongside ranking signals:

- Activity
- Health
- Visibility
- Growth

Ranking is deterministic. For the same profile, same platform filters, same dataset, and same feedback state, pressing "Refresh Rank" should return the same ordering. Rankings change when the user changes personas, edits a custom profile, changes platform filters, adds feedback, or ingests new live records. The dashboard shows a "Last ranked" timestamp so a refresh is visible even when the deterministic order stays the same.

## Explainability

Every ranked card includes a "Why this?" explanation generated from the scoring features. Example explanation elements include:

- domain match
- semantic relevance
- community health
- visibility
- effort estimate
- good-first-issue signal
- rising-fast signal

Suggested actions are template-generated from the source and opportunity type. For example, GitHub opportunities suggest opening an issue or submitting a small PR, while Forem (DEV.to) opportunities suggest concise discussion comments.

## Adaptive Learning

Adaptive learning is implemented through in-app feedback:

- `engage`
- `bookmark`
- `skip`

Feedback updates tag-level weights. Positive actions increase affinity for similar tags, while skips decrease affinity. Future recommendations include this feedback affinity in the final score.

The live dashboard persists real user feedback to:

```text
data/feedback_events.json
```

Each event stores the timestamp, action, opportunity id/source/domain/title/tags, and the active profile name, interests, goal, and platforms. When the app starts, `code/engageiq/feedback_store.py` replays saved events into the ranker so learned tag weights survive dashboard restarts.

The simulation in `code/engageiq/learning.py` runs 60 feedback rounds and reports Precision@10 before and after feedback. This demonstrates measurable adaptation while keeping the learning logic explainable for a live demo.

## Privacy Position

EngageIQ ranks public engagement opportunities. It does not track a user's private activity on GitHub, DEV.to, Hacker News, or other platforms. The app does not read private messages, saved posts, votes, private repositories, browsing history, or personal account analytics. The only user-specific data stored by the dashboard is local in-app profile input and feedback. Saved custom profiles are stored in the browser's `localStorage`, and feedback clicks are stored locally in `data/feedback_events.json`.

## Current Limitations

This build is optimized for a reliable offline grading demo. The main limitations are:

- The dataset is synthetic but schema-compatible with public API data.
- Live collectors exist for GitHub, GH Archive, Hacker News, and Forem (DEV.to), but the dashboard primarily uses the offline snapshot unless `code/ingest_live.py` is run.
- The pipeline is batch/offline rather than a true continuously running stream.
- The first SBERT run requires downloading the model unless it is already cached.
- Suggested actions are template-generated rather than LLM-generated.
- Batch analytics use pandas rather than Spark or another distributed framework.

These are suitable points to disclose in the technical brief under limitations and future work.
